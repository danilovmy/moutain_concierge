
2. “Лист хотелок” (MVP)

MVP “Горный Консьерж” – CRM-прототип

1. Роли
– Админ/консьерж (одна роль на старте, логин/пароль).

2. Партнёры
– Сущность Partner.
– Поля: тип (инструктор/прокат/отель/такси/няня/SPA/другое), название/ФИО, телефон, WhatsApp, email, курорт/регион, комментарий, статус (активен/на паузе).
– Список партнёров (таблица) + форма создания/редактирования.

3. Лиды (заявки гостей)
– Сущность Lead.
– Поля: дата создания, имя гостя, контакт (телефон/WhatsApp), источник (отель/прокат/такси/сайт/другое + текст), что нужно (чекбоксы: инструктор/прокат/скипасс/отель/такси/няня/SPA/другое), даты поездки (с/по), статус (новый/в работе/офер/забронировано/отменён/нет ответа), комментарий.
– Таблица лидов с фильтрами по статусу/источнику/дате.
– Карточка лида (просмотр/редактирование).

4. Брони (Bookings)
– Сущность Booking, создаётся из лида.
– Поля: дата создания, ссылка на Lead, гость (имя), дата/даты услуги, статус (черновик/подтверждена/выполнена/отменена/no-show).
– Внутри брони — список услуг (line items): тип услуги (инструктор/прокат/такси/няня/SPA/другое), партнёр (ссылка на Partner), краткое описание, цена для гостя, поле “наша комиссия” (числом, пока ручной ввод).
– Таблица броней с фильтрами по дате/статусу/партнёру.

5. Мини-отчётность
– Простая таблица/страница:
– фильтр по партнёру и периоду,
– список его броней,
– сумма оборота,
– сумма нашей комиссии (по полю из line items).

6. Технически
– Стек Python Django, фронт хоть на чистом html (vue in future)
– Нужен веб-интерфейс, адаптив под мобилу будет плюсом.
– Без интеграций с почтой/WhatsApp, без напоминалок — всё это на второй этап.
– Код в отдельном репо (GitHub/GitLab), чтобы потом можно было доращивать.
